# IPA Typing Keyboard — v1.1 roadmap

Items deliberately deferred from v1 shipping scope. Revisit when planning v1.1.

## Long-press UX

### Haptic feedback intensity / variety

v1 wires `UISelectionFeedbackGenerator` at three points in the long-press flow
(popover appear, drag-snap to a new variant, release on a variant). On a
keyboard extension with `RequestsOpenAccess = NO`, this is the only feedback
generator type that fires reliably.

What we cannot do without flipping `RequestsOpenAccess` to `YES`:
- `UIImpactFeedbackGenerator` (light / medium / heavy taps) — Apple gates these
  behind Full Access in extensions. Silently no-ops otherwise.
- `UINotificationFeedbackGenerator` (success / warning / error) — same gate.
- `AudioServicesPlaySystemSound` for haptic sound IDs (1519/1520/1521) — same.

**Why not flip Full Access:** the entire privacy story (spec §5.4, §9.3,
PrivacyInfo.xcprivacy, App Store Connect "Data Not Collected" label) rests on
the extension having no Full Access. Flipping it triggers a user permission
prompt, changes the privacy label, and requires updating
`docs/ios/review-notes.md`. Not worth it for richer haptics alone.

### Popover that extends above the keyboard frame

v1 clamps the popover to `keyboardBounds.minY` for top-row keys, so the popover
overlaps the long-pressed key. Apple's built-in keyboard renders the popover
*above* the keyboard frame (in the host app's display area).

**Why this isn't a SwiftUI limitation specifically:** `UIInputViewController.view`
has a fixed frame given by UIKit. Anything drawn inside — UIKit views, SwiftUI
views, raw CALayers — gets clipped to that frame. To render above, the only
real path is a **separate `UIWindow`** attached to the keyboard's `windowScene`,
positioned above the input view in screen coordinates. SwiftUI works fine
inside that new window (via `UIHostingController`); the SwiftUI choice doesn't
cost us anything here, the UIKit plumbing is the cost.

Reference apps that ship this: Gboard, SwiftKey, Grammarly Keyboard.

**v1.1 implementation sketch:**

1. Subclass `UIWindow` as `PopoverWindow`. On init, take the
   `UIInputViewController`'s `view.window?.windowScene` and bind to it.
2. Set its frame to a portion of the screen above the keyboard's input view
   (use `UIScreen.main.coordinateSpace` to convert).
3. `windowLevel = .alert + 1` so it sits above the keyboard's own window.
4. Host the existing `VariantPopover` SwiftUI view inside via
   `UIHostingController(rootView: VariantPopover(...))`.
5. Show/hide the window synchronously with the `popoverKey` state in
   `KeyboardRootView` — bridge with `@StateObject` or a `Coordinator`.
6. Tear the window down on `viewWillDisappear` and on the
   `.ipaKeyboardShouldCancelGesture` notification.

Edge cases to test before submission:
- Split keyboard / floating keyboard on iPad
- Keyboard rotation
- Background → foreground → keyboard reappear
- Other extension keyboards switching in mid-gesture

**Decision for v1:** ship with "clamp to top". Revisit if external-beta feedback
flags the overlap as confusing.

### Slide-out velocity / animation curve tuning

v1 uses `.easeOut(duration: 0.12)` for popover appear and `.easeOut(duration:
0.1)` for dismiss. Apple's keyboard uses a custom spring with overshoot.
Matching that exactly requires reverse-engineering or accepting an
approximation; not on the critical path for v1.

## Other deferrals

### Phase 5 device-only verification

The "tap Open Settings → lands on the app's Settings page" smoke test was
deferred in Phase 5 (pre-signing-setup). After build 2 is on TestFlight, this
becomes a one-line check during the internal smoke pass — close it then.

### App icon source / variants

v1 reuses the companion-app Tauri icon (`companion-app/src-tauri/icons/icon.png`)
flattened onto black. Single-size 1024×1024 universal entry; `actool`
synthesizes device variants at build time.

For v1.1 consider:
- A dedicated iOS icon with the curved-corner aesthetic baked for visual review
- Dark / tinted variants (iOS 18+ supports `dark` + `tinted` AppIcon images)
- A different visual that distinguishes the iOS app from the Tauri companion app

### Localization

v1 ships English only. Symbol names are bundled (`LocalizedSymbolNames.swift`)
in English. Phase 1 of v1.1 localization would translate the Setup tab copy
and the symbol reference names into additional languages.

### Snapshot test baseline regeneration

v1 build 2 changed `LayoutEngine.popoverRect` for top-row keys, which
invalidates the `test_popover_rightEdge_o_topRow_iPhone13_light` snapshot.
Re-record locally (`isRecording = true` in `PopoverSnapshotTests.setUp`, run
the suite once, set back to false) before relying on snapshot tests in CI.
